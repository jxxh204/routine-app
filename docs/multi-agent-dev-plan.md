# routine-app 멀티 에이전트 운영안 (Backend / Frontend / Designer / QA)

## 0) 핵심 원칙 (필수)

- 모든 작업은 **TDD** 기반으로 진행 (Red → Green → Refactor)
- 기능 완료 기준은 "코드"가 아니라 **검증 가능한 동작 + 책임자 사인오프**
- PR 머지 전, 4개 에이전트 체크리스트를 모두 통과해야 함

---

## 1) 에이전트 구성 (4개)

### A. Backend Agent (BE)
**책임 범위**
- API 계약(요청/응답/에러코드) 정의 및 유지
- DB 접근 경로/API 보안(인증/인가/RLS 영향) 보장
- 타임존/정합성/멱등성(idempotency) 처리
- 서버 로그/에러 핸들링/운영 가드레일

**산출물**
- API 변경 요약
- 데이터 정합성 테스트(단위/통합)
- 보안 체크 결과

---

### B. Frontend Agent (FE)
**책임 범위**
- 사용자 플로우 구현 (입력→저장→조회까지)
- TanStack Query 캐시 정책(optimistic/invalidate/staleTime)
- 로딩/에러/빈 상태 처리
- 반응형/성능/기본 접근성

**산출물**
- 화면 동작 PR
- 핵심 플로우 테스트(컴포넌트/통합)
- 캐시 무효화/동기화 설계 근거

---

### C. Designer Agent (DS)
**책임 범위**
- UX 상태 모델 정리(기본/로딩/성공/실패/빈 상태)
- 디자인 일관성(토큰, 간격, 컴포넌트 사용 규칙)
- 마이크로 인터랙션(스켈레톤, 피드백 문구)
- 모바일 기준 시각 검수

**산출물**
- 상태별 UI 명세
- 스크린샷 기준(모바일) 및 검수 코멘트

---

### D. QA Agent (QA)
**책임 범위**
- 회귀 시나리오 관리(핵심 경로 우선)
- E2E/통합 테스트 실행 및 실패 원인 분류
- 재현 절차 문서화
- 릴리즈 전 최종 게이트 판단

**산출물**
- 테스트 실행 로그
- 실패/리스크 리포트
- 릴리즈 GO/NO-GO 판정

---

## 2) 책임 매트릭스 (RACI)

| 작업 | Backend | Frontend | Designer | QA |
|---|---|---|---|---|
| API 계약/보안 | A/R | C | I | C |
| 데이터 정합성(시간대/중복/멱등) | A/R | C | I | C |
| 화면 구현/상호작용 | C | A/R | C | C |
| 로딩/스켈레톤/오류 UX | I | R | A | C |
| 캐시 전략(query/invalidate) | C | A/R | C | C |
| 테스트 설계(단위/통합/E2E) | C | C | I | A/R |
| 릴리즈 판정 | C | C | C | A/R |

---

## 3) 핸드오프 계약 (Agent → Agent)

### BE → FE
- API 계약 확정 전 FE 구현 금지
- 응답 예시(JSON) + 에러코드표 제공 필수

### FE → DS
- 상태별 스크린샷(기본/로딩/성공/실패) 첨부 필수
- 문구/피드백 타이밍 정의 필수

### DS → QA
- 검수 기준(정상/비정상 케이스) 전달 필수

### QA → Merge
- 재현 시나리오 PASS 로그 + 잔여 리스크 명시

---

## 4) Merge Gate (필수 8항목)

머지 전 아래 전부 충족:
- [ ] BE: 인증/인가/쿼리 경로 점검 완료
- [ ] BE: 타임존/멱등성/중복저장 케이스 검증
- [ ] FE: optimistic + invalidate 동기화 확인
- [ ] FE: 로딩/에러/빈 상태 UI 존재
- [ ] DS: 모바일 기준 스크린샷 검수 완료
- [ ] QA: 핵심 플로우 E2E PASS
- [ ] QA: 회귀 테스트 PASS
- [ ] PR 본문에 리스크/롤백 절차 명시

---

## 5) 핵심 회귀 시나리오 (루틴앱 고정)

1. 오늘 루틴 인증 직후 캘린더 오늘 날짜에 즉시 반영
2. 새로고침 후에도 캘린더/투데이 정합성 유지
3. KST 자정 경계에서 날짜 어긋남 없음
4. 네트워크 실패 시 사용자 피드백 + 재시도 가능
5. 친구/본인 데이터 분리 및 권한 누수 없음

---

## 6) 운영 방식 (OpenClaw)

- 작업 착수 시 4개 서브에이전트 역할로 분담
  - `backend-agent`
  - `frontend-agent`
  - `designer-agent`
  - `qa-agent`
- 메인 세션은 조율/우선순위/최종 승인만 담당
- 각 에이전트 결과를 PR 코멘트로 남겨 책임 추적 가능하게 유지

---

## 7) 성장형 에이전트 운영 규칙 (직무 집중 + 스킬 자산화)

각 에이전트는 **자기 직무 외 판단/수정은 하지 않고**, 필요 시 해당 직무 에이전트에 명시적으로 핸드오프한다.

- Backend Agent: API/DB/보안/정합성 외 영역 직접 수정 금지
- Frontend Agent: UI/상태관리/캐시/상호작용 외 영역 직접 수정 금지
- Designer Agent: UX 상태/컴포넌트 사용성/피드백 문구 외 영역 직접 수정 금지
- QA Agent: 검증/재현/게이트 외 영역 직접 수정 금지

### 스킬 자산화 원칙

반복 업무/재발 이슈는 반드시 스킬로 승격한다.

1. 트리거
- 같은 유형 이슈 2회 이상 재발
- 수동 점검이 10분 이상 반복
- 사람 의존 체크리스트가 자주 누락

2. 스킬화 대상 예시
- 인증 직후 캘린더 반영 회귀 점검
- KST 날짜키/자정 경계 검증
- TanStack Query invalidate 정책 점검
- 배포 전 API 캐시 헤더/no-store 검증

3. 스킬 생성 규칙
- 위치: `skills/<role>-<topic>/SKILL.md`
- 포함 항목: 목적, 입력, 실행 절차, 실패시 분기, 완료 기준, 산출물
- 각 PR에서 새 교훈이 나오면 관련 스킬을 즉시 업데이트

4. 역할별 책임
- Backend: API 정합성/보안 스킬 유지
- Frontend: 캐시/로딩/상태전이 스킬 유지
- Designer: 상태 UX/마이크로카피 스킬 유지
- QA: 회귀/E2E/릴리즈 게이트 스킬 유지

---

## 8) 즉시 적용 체크리스트

- [ ] 본 문서를 팀 운영 기준으로 채택
- [ ] 다음 이슈부터 4-agent 분업으로 수행
- [ ] PR 템플릿에 Agent 체크박스 추가
- [ ] 반복 이슈를 월 1회 스킬로 승격/정리
- [ ] 1주일 운영 후 누락/병목 재조정
